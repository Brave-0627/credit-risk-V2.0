# 一、先做一件关键的事：技术 → 前端功能映射

你的可行性报告本质上已经定义了系统能力，可以抽象为4层：

| 技术层（报告）            | 系统能力             | 前端对应模块            |
| ------------------------- | -------------------- | ----------------------- |
| 数据采集/治理             | 数据录入、清洗、管理 | 数据输入页 / 数据管理页 |
| 数据表征（画像）          | 用户/企业画像        | 画像展示页              |
| 风险预测模型（ML/大模型） | 风险评分/分类        | 预测结果页              |
| 应用示范系统              | 决策支持、推送       | 推荐/监控/预警页        |

👉 结论：
**前端不是围绕“算法”，而是围绕“数据流 + 决策流”设计**

# 三、四类用户的前端结构设计（核心部分）

## 1️⃣ 个人用户端（重点）

### 功能结构

```
个人首页（Dashboard）
├── 我的信用画像
├── 风险预测
├── 历史记录
├── 数据管理
```

### （1）信用风险预测页面（核心页面）

👉 本质：**输入 → 模型 → 输出**

#### 输入（对应报告中的“多源数据” ）

你要做成“表单 + 可扩展”：

- 基本信息
  - 年龄、职业、收入
- 财务数据
  - 负债、资产、流水
- 行为数据
  - 消费记录、借贷记录
- 非结构化（高级）
  - 文本（可选）

👉 UI建议：

- 左侧：输入表单（分模块折叠）
- 右侧：实时提示（数据完整度）

------

#### 输出（核心展示）

必须可解释（报告强调“解释性”）

建议展示：

- 风险等级（A/B/C/D）
- 风险概率（如：23%）
- 关键影响因素（TOP 5）
- 建议（如：降低负债率）

👉 UI组件：

- 仪表盘（Gauge）
- 雷达图（画像）
- 条形图（因素贡献）

------

### （2）用户画像页面

对应报告中的：

> “多维主体画像技术”

展示：

- 信用评分维度分解
- 行为特征标签（如：稳定型、风险型）
- 时间趋势（信用变化）

### 核心功能

#### （1）模型上传

- 上传模型文件（如.pkl / API）
- 填写：
  - 输入格式
  - 输出格式
  - 模型说明

👉 这一步就是你问的“输入输出在哪体现”——就在这里定义

------

#### （2）模型调用配置

系统要做一件关键事：

👉 **统一模型接口（标准化输入输出）**

```json
输入：
{
  "income": 10000,
  "debt": 5000
}

输出：
{
  "risk_score": 0.23,
  "level": "B"
}
```

# 四、全系统最关键的一张图（你必须理解）

## 数据流（决定界面）

```
用户输入 → 数据处理 → 模型预测 → 结果展示 → 决策应用
```

对应前端：

| 流程 | 页面        |
| ---- | ----------- |
| 输入 | 表单页      |
| 处理 | （后端）    |
| 预测 | 按钮触发    |
| 输出 | 结果页      |
| 应用 | 推荐/报告页 |

# 七、关键提醒（避免你走弯路）

### ❌ 常见错误

- 直接围绕“算法”设计页面（用户看不懂）
- 输入字段随便写（导致模型无法接）
- 没有统一输入输出格式（科研模块会崩）

------

### ✅ 正确思路

- 前端 = 数据入口 + 结果解释

- 算法 = 黑盒（但要可解释输出）

- 所有功能围绕：

  > “用户如何输入 → 系统如何判断 → 用户如何理解结果”
